# 拷问集成规范

将 $grilling 作为工作流的决策门禁，而不是泛化的问答步骤。

## 触发后顺序

1. 读取当前阶段的项目配置、交接物、决策日志、测试证据和实现事实。
2. 区分已验证事实与需要人工选择的取舍；不要重复询问已有明确结论。
3. 调用 $grilling，一次只提出一个问题。每题必须说明问题、影响、推荐答案和推荐理由。
4. 用户回答出现冲突或新依赖时，优先拷问该冲突。未获得“已达成共同理解”或等效确认前，不修改计划、需求、设计、代码、资源或发布状态。
5. 确认后更新 docs/grilling-log.md、decision-log.md 和受影响交接物，并将变更影响、未决项、拒绝项和下一质量门写入 handoff-status.md。

## 需求变更处理

将功能新增、功能删改、优先级调整、验收变化、渠道变化、能力开关变化、技术路线变化、数值目标变化、资源与视觉方向变化、质量指标变化和发布风险豁免均视为需求修改。

先冻结受影响范围，记录变更来源和影响面；完成 $grilling 后才更新基线。若变更影响已通过的质量门，标记该质量门为待复核，并只重跑受影响的验证项。

## 记录格式

docs/grilling-log.md 的每条记录至少包含日期、阶段、触发原因、已核实事实、问题、用户回答、推荐项、共同理解确认、受影响文档/角色、下一步和未决项。不要记录凭据、个人数据或受限合同全文。
