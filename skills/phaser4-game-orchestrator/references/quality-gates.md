# 质量门与并行边界

## G0：立项批准

人工必须确认：核心循环、最小范围、首发渠道、能力开关、资源来源方案、预算负责人和未知指标的决策截止点。

允许并行：技术可行性、数值可行性、资源规格、音频方向。

阻断条件：渠道不明、核心体验未定义、商业能力未经批准、资源权属无路径。

## G1：垂直切片

必须有一个从启动到结算的核心循环，可从干净环境构建；资源可追溯；至少一个选定目标可运行；测试计划含当前已知设备和指标缺口。

允许并行：新增玩法、数值迭代、资源替换、性能基线采集。

阻断条件：不能构建、核心循环无验收、资源权属不明、平台适配无法运行。

## G2：发布候选范围冻结

必须有冻结清单、目标设备测试证据、性能证据、P0/P1 缺陷处置结论、数据与存档策略、资源清单和渠道差异结论。

允许并行：最终回归、商店素材准备、候选包构建、合规资料整理。

阻断条件：质量指标仍为空、阻断缺陷未处置、渠道要求不完整、候选包不可复现。

## G3：人工发布放行

必须有逐渠道候选包、提审资料、版本号、发布回滚或热修复方案、已知风险和人工签署结论。

阻断条件：没有人工放行、凭据管理不安全、隐私或授权未确认、构建与测试证据不对应。

## 并行 worktree 收敛

每个质量门前，已完成的并行分支必须完成自动收敛或被明确标为阻断。合并前要求基线和角色 worktree 均干净、分支有提交与检查证据、交接状态可验收；合并后执行 Git 空白错误检查，随后自动删除本地 worktree 和已合并分支。

阻断条件：合并冲突、未提交改动、检查失败、缺少证据、待人工决策或工作项超出已批准范围。阻断时保留分支与 worktree，禁止强制删除。
