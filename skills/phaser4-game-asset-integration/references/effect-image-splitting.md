# 效果图拆分为 Phaser 单图

把效果图中的可复用视觉元素转换成可直接加载、定位、打包和验证的独立资源。效果图不是最终资源，裁出矩形截图也不等于完成拆分。

## 拆分前审查

1. 读取 GDD、TDD、效果图和资源登记。优先按“来源内容哈希 + 元素编号”生成稳定且唯一的拆分 ID；无法计算哈希时才使用“规范化来源路径 + 明确版本 + 元素编号”。来源文件移动或改名但内容未变时应命中原记录。再生成带编号的效果图标注预览与拆分清单，逐项写明用途、元素边界、目标像素尺寸、缩放倍率、透明需求、锚点、九宫格或动画帧要求、交付形式，以及需要补绘或重新生成的区域。
2. 按拆分 ID、来源版本和规格查询已有记录。已有产物同时满足文件存在、纹理键可寻址且最近一次 Phaser 验证有效时，将其标为“复用”并排除出本次拆图；记录缺失、文件失效或规格变化时标为“待修复”或“待重拆”，不得因为文件名相似而盲目复用。
3. 审查清单是否覆盖全部运行时元素，是否存在遗漏、重复、错误合并或无运行时用途的碎片；确认角色、部件、道具、地形、背景层、UI、特效和状态变体的拆分粒度符合玩法与复用需求。
4. 审查源图可拆性：检查遮挡、透视变形、光照烘焙、背景污染、分辨率和边缘信息是否足以形成完整单图。信息不足的项目必须明确补绘或重新生成方案，不得直接裁切。
5. 审查 Phaser 约定：确认文件名、纹理键和帧名唯一，动画画布与基线一致，锚点、九宫格、图集归组及移动端预算可执行。
6. 将审查结果标为“通过”“待修改”或“阻断”，记录审查人或验证角色、日期和证据。仅“通过”的新拆或重拆项目可以进入拆分；改变玩法边界、视觉方向、成本或发布风险的争议提交总控决策，其他问题修订后重新审查。

拆分清单、编号预览、来源版本或规格发生变化时，受影响项目必须生成新修订记录并重新审查；旧产物标为“已取代”，不要静默覆盖。不得以拆分后的成品反向替代拆分前审查。

## 拆分与导出

1. 沿真实轮廓分离元素，移除背景、文字标注、选框、阴影污染和相邻元素；保留业务要求的自身阴影与发光。
2. 为被遮挡元素补全运行时会显露的区域。成组动画帧使用一致画布、基线、朝向和锚点，避免播放抖动。
3. 默认输出透明 PNG；仅对确定不需要透明的完整背景使用 JPEG 或 WebP。不要仅为“整齐”强制拉伸，也不要把非正方形资源填充成正方形。
4. 裁掉无效透明区域，同时为滤波、旋转或发光保留足够边距。进入图集的资源使用至少 2 px 挤出或等效防溢色设置，避免线性采样串色。
5. 使用稳定的小写短横线文件名；同一状态或动画采用连续、可排序的命名。按 TDD 输出独立文件或图集，并提供可直接使用的 Phaser 纹理键、帧名和默认锚点。

## 必须交付

- 拆分前审查记录：带编号的效果图标注预览、包含拆分 ID 与来源版本的拆分清单、复用/新拆/重拆结论、审查证据以及待补绘或重新生成项。
- 独立图片或图集图片及对应 atlas JSON；不得只交效果图、切图坐标或编辑器工程文件。
- 每个资源的来源效果图、用途、像素尺寸、格式、透明方式、默认锚点、纹理键/帧名和文件路径记录。
- 对比预览：在棋盘格与深浅两种底色上展示逐图结果，便于发现白边、黑边、残留背景、截断和空白过大。
- Phaser 验证证据：实际通过 `this.load.image`、`this.load.atlas` 或对应加载方式载入，在目标缩放、相机和渲染设置下显示正确。

## 验收门槛

逐项确认后才能标记“可交付”：

- 审查有效：对应拆分清单在开工前已通过审查，且拆分期间没有未经复审的规格或范围变化。
- 去重有效：每个拆分 ID 只有一个当前有效产物；已验证产物被复用，重复文件、重复纹理键与已取代版本未进入运行时资源。
- 画面完整：无误裁、残影、非预期背景、边缘缺口、透明杂点和可见接缝。
- 规格正确：像素尺寸、方向、状态、帧序、锚点、九宫格边界和命名与 TDD 一致。
- 渲染可靠：浏览器控制台无加载错误；纹理键与帧名可寻址；线性采样、缩放和旋转时无明显溢色或抖动。
- 性能合格：无重复纹理和无效大画布；图集尺寸、纹理数量、显存与包体符合已批准预算，并在至少一台目标设备验证。
- 可追溯：来源、生成或编辑记录、授权、责任人及发布资格已经写入资源登记。

任何一项失败都将该资源标为“待修复”，记录复现截图或运行步骤；不要用“接入时再处理”代替拆分验收。完成后把状态更新为“已拆分”并写回实际路径、纹理键和验证证据，确保后续任务能够查重。
