# Grilling

在任何决策型任务前或执行中，沿着决策依赖逐项澄清。只让用户决定不能从代码库或现有文档确认的事项；不要阻塞纯事实查询或已确认的低风险执行。

## 进入方式

1. 先识别当前任务所属阶段，并检查该阶段的计划、设计、技术方案、验收记录和代码库事实。
2. 无 PRD 时，进行 PRD 前拷问：从目标用户、问题、首次价值、范围、商业约束和风险开始。
3. 存在任一 PRD 文件时，完整阅读已有 PRD，并围绕其中的缺口、冲突、假设和依赖进行拷问；不要重复询问已经明确记录的事实。
4. 在 UX/UI、技术方案、实施计划、交付或发布阶段，围绕当前阶段尚未确认的范围、优先级、取舍、风险、验收和依赖进行拷问。
5. 每个模块首次进入实现阶段时必须重新进入 grilling。先读取 PRD、模块图、技术设计、现有实现和已确认决策，再围绕该模块包含的用户功能、明确非目标、页面与状态边界、跨模块依赖和模块验收路径逐项确认；不要在确认完成前细化页面功能或生成任务简报。
6. UI 模块完成功能和页面细化后、生成任何效果图前，再执行一次模块效果图拷问。确认模块视觉目标、需要效果图的页面和状态、页面预算、签名要求、可接受的自定义组件/动效/位图/维护成本，以及效果图不得改变已确认范围。每个模块执行一次；只有出现新的视觉或成本决策时才对页面追加追问。
7. 全局视觉冻结只确认方向定位、视觉签名、实现成本、表达预算覆盖项及最终冻结意愿；不得生成或评审页面效果图。仅当审计发现新的或冲突的决策时重新进入完整 grilling。
8. 先用代码库、已有文档、配置和证据确认可验证的事实；只有产品取舍、优先级、接受的风险和品牌方向属于用户决策。
9. 除模块首次进入实现及模块效果图拷问这两个强制节点外，若不存在需要用户决定的事项，明确说明无需进入 grilling，并继续原任务。

## 对话规则

- 一次只问一个问题，收到回答后再继续。
- 每个问题必须包含：问题、该决策影响、推荐答案及其理由。
- 从根决策到依赖决策逐层推进；只展开与当前范围相关的分支。
- 若用户回答暴露冲突或新依赖，优先追问该冲突，再回到原分支。
- 不把推荐答案伪装成既定事实；用户可以选择其他方案或拒绝该推荐。
- 用户未明确确认“已达成共同理解”或等效表述前，不执行、修改或推进被拷问的计划。

## 决策树

按需要依次覆盖以下分支：

1. 产品价值：目标用户、问题、差异化、首次价值和成功标准。
2. 范围：MVP、非目标、优先级、依赖和可接受的延期项。
3. 尝试与信任：注册、付费、权限、隐私、数据处理、可撤销性和恢复路径。
4. 体验：关键用户流、页面状态、失败处理、反馈和无障碍要求。
5. 设计：产品气质、信息层级、设计系统、装饰边界和可验证的视觉质量标准。
6. 交付：技术约束、验证平台、发布风险、证据和验收条件。
7. 模块实现：模块要实现的用户结果、包含功能、非目标、页面与状态边界、跨模块交互、完成定义和可延期项。
8. 全局视觉冻结：方向定位选择、可复述视觉签名、表达预算覆盖、实现成本承诺和冻结验收；不包含效果图。
9. 模块效果图：模块视觉目标、页面和状态清单、页面预算、签名强度、资产与实现成本、范围一致性和完成定义。

## 完成与交接

在用户明确确认共同理解后，汇总已确认的决策、未决项、依赖、拒绝的选项和后续阶段。对于 Phaser 4 游戏工作流，在 docs/grilling-log.md 记录这些内容及用户确认，并同步 decision-log.md 与 handoff-status.md。其他产品研发工作流在 docs/product/grilling-log.md 记录这些内容。模块实现阶段将功能范围确认和模块效果图拷问分别记录，并同步到 docs/plans/modules/<module-name>-scope.md；只有两道门禁都通过，才可生成当前模块的页面效果图。

## 禁止事项

- 不一次提出多个问题。
- 不询问可通过探索代码库确认的事实。
- 不在用户确认前生成 PRD、设计稿、技术方案、代码或实施计划。
- 不在模块级共同理解确认前细化该模块的页面功能、创建实现任务简报或开始设计与代码实现。
- 不在模块效果图拷问确认前生成当前模块的任何页面效果图。
- 不在全局视觉定位阶段生成代表页面、模块或页面效果图。
- 不把未回答的问题当成默认同意。
