# Phaser 4 技术架构

## 职责

建立可维护、可测试、可构建的技术边界，并让小游戏、iOS、Google Play 的差异显式化。开始实现前读取 docs/GDD.md、docs/project-profile.yaml、docs/platform-matrix.md 和 docs/decision-log.md。

技术路线、平台适配、依赖、质量指标或验收边界需要人工取舍，或这些内容发生需求修改时，先调用 $grilling 并等待人工确认共同理解；不要先创建或修改技术实现。

## 技术基线

- 以 Phaser 4、TypeScript 和 Vite 实现游戏 Web 核心。需要创建项目时，优先使用官方当前的 create-phaser-game 流程，不要固定过期模板或版本号。
- 使用 Capacitor 生成 iOS 和 Android 原生容器；将容器插件封装在平台适配层，禁止业务玩法直接依赖原生 API。
- 为小游戏建立独立可行性验证与适配清单。不要假设 Web 构建可以不经验证直接发布到小游戏渠道。
- 将可选商业能力做成明确接口与开关；未批准能力应使用空实现或完全不接入，不能留下隐式线上调用。

## 执行流程

1. 在 TDD.md 记录运行时版本、目录职责、场景生命周期、游戏状态、输入、资源加载、存档、错误处理与构建命令。
2. 规划逻辑、表现、平台和第三方服务的边界，避免把核心规则散落在场景回调中。
3. 为横竖屏、刘海安全区、触控、暂停恢复、后台切换和网络不可用定义验证策略。
4. 完成最小项目骨架与可复现生产构建后，把命令和证据写入 TDD.md 与 handoff-status.md。
5. 将不确定的渠道能力、性能指标或原生插件选择写入 decision-log.md，等待人工决定。

## 验收与交接

交给玩法开发的架构必须说明模块入口、数据流和测试入口；交给测试性能的架构必须提供构建命令、版本和日志定位方式。新增或修改的类、函数、实体定义均添加简体中文注释，解释不直观的设计边界。
